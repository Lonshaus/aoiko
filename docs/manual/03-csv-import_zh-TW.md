# 03. CSV 匯入

從銀行・信用卡的交易紀錄 CSV 一次建立傳票。

**Language**: [日本語](03-csv-import.md) | [English](03-csv-import_en.md) | **繁體中文**

> **讀完這章你能做到**
> - 抓對應銀行・信用卡的 CSV 然後匯入
> - 看懂重複檔案偵測怎麼運作
> - 用自動分類規則讓對應科目自動填入
> - 用 LLM 自動分類
> - 從匯入紀錄整批沖銷
>
> **前提**：[01. 初次設定](01-setup_zh-TW.md) 做完。要用卡片別輔助科目（例 `2120 未払金 / 楽天卡`）的話也要先登記。

## 1. 支援的銀行・信用卡

| 種類 | 支援 |
|---|---|
| 銀行 | 三菱UFJ／三井住友／SBI新生／PayPay（信用卡決済運用、餘額未對應）|
| 信用卡 | 楽天／JCB（含 Recruit Card 等）／セゾン／三井住友／三菱UFJ／au PAY／PayPay／ビュー（JRE CARD）／ライフ |

> 全部都用真實 CSV 驗證過。要新增銀行・卡片 parser 請看 [CONTRIBUTING.md](../../CONTRIBUTING.md)。

## 2. 抓 CSV

從各家會員網站抓 CSV。各家入口指引：

| 服務 | 抓 CSV 路徑 |
|---|---|
| 三菱UFJ 銀行 | ダイレクト 登入 → 入出金明細 → 期間指定 → CSV 下載 |
| 三井住友銀行 | SMBC ダイレクト → 入出金明細 → CSV |
| SBI新生銀行 | パワーダイレクト → 入出金照会 → CSV |
| 楽天卡 | e-NAVI 登入 → 利用明細 → CSV 下載 |
| JCB / Recruit Card | MyJCB → ご利用明細 → CSV |
| セゾンカード | Net Answer → 利用明細 → CSV |
| 三井住友カード | Vpass → 明細 → CSV |
| 三菱UFJカード | 會員網站 → 明細 → CSV |
| au PAY カード | 會員網站 → 利用明細 → CSV |
| ビューカード | VIEW's NET → 利用明細 → CSV |
| ライフカード | LIFE-Web Desk → 利用明細 → CSV |

> **PayPay カード**：會員網站動線會變、有些帳號 Web 給不出 CSV，只能 OCR 或手動。
>
> **編碼**：parser 自動判斷（Shift_JIS / UTF-8）。如果亂碼請確認來源的輸出編碼。

## 3. 匯入 3 步驟

導覽列**「匯入」**進入 `Import` 畫面。

### 3-1. 步驟 1：選 parser

**「1. 解析器」**下拉選來源。選了之後**「已知側科目」**會自動帶出（例：`楽天卡` → `2120 未払金`）。

如果有用輔助科目分多個帳戶／卡片、再選**「輔助科目（選填）」**對應的（例：`楽天卡`）。

> 已知側科目是 CSV 每列共通的傳票一邊（卡片＝未払金、銀行＝普通預金）。**對應科目**（經費・銷貨等）下一步逐列選。

### 3-2. 步驟 2：選 CSV 檔案

**「2. CSV 檔案」**選檔。

選好之後 aoiko 會自動讀內容、下方顯示預覽。

#### 重複偵測

aoiko 會記每個檔案的**檔案 hash**（內容指紋）。再次匯入相同檔案會跳紅色 banner：
> 這個檔案在 {date} 已經匯入過了（{name}）

防止誤匯入重複。

> 重新下載相同期間的 CSV 內容會相同所以偵測得到。期間部分重疊的不同 CSV（例：上月底〜本月底 vs 本月）hash 不同抓不到、要自己確認重複列。

### 3-3. 步驟 3：為各列選對應科目

CSV 每列以**匯入候選**形式表格化。每列：

| 欄位 | 內容 |
|---|---|
| 日期 / 摘要 / 金額 | 從 CSV 讀進來的 |
| 對應科目 | 這列借方／貸方的**另一邊**（你選）|
| 輔助科目（選填）| 對應科目的輔助科目 |
| 跳過 | 勾起來就不匯入這列 |

#### 自動填入徽章

表格上方會顯示**「{valid} / {total} 筆預定登錄」**、各列可能會出現以下徽章：

| 徽章 | 意思 |
|---|---|
| **規則** | 命中自動分類規則（見 [6 節](#6-自動分類規則)）|
| **LLM↑** | LLM 高信度推測對應科目 |
| **LLM↓** | LLM 推測但信度低、建議確認 |
| （無徽章）| 沒有自動填、需要手動選 |

每個徽章 hover 看詳細。LLM 徽章只在 LLM 連動有設的時候才出現。

#### 一覽確認

- **規則**徽章基本上信任不動
- **LLM↑** 看一眼確認妥當
- **LLM↓** 一定要確認
- 沒徽章的列**對應科目**自己選

### 3-4. 登錄

按**「建立 {count} 筆」**會一口氣不確認直接全部變傳票。

> 按之前數量跟徽章好好確認。錯了可以用沖銷傳票補救但量大時很累。
>
> 大量誤登錄發現的時候、從導覽列**「匯入紀錄」**做**整批沖銷**（[5 節](#5-匯入紀錄與整批沖銷)）比較有效率。

成功時上方會顯示**「已建立 {count} 筆傳票」**。

## 4. LLM 自動分類（選用）

在 [01. 初次設定 § 7](01-setup_zh-TW.md#7-要用-ocrllm-才做的設定) 設好 Gemini API 金鑰或 OpenAI 相容 endpoint，CSV 匯入時：

- **沒命中規則的列**送 LLM 依文脈推測、對應科目預填
- 徽章**「LLM↑」** / **「LLM↓」**標示信度

> **送什麼**：CSV 列文字（金額・摘要等）＋ 勘定科目一覽
> **送哪**：選中的引擎（Gemini → `generativelanguage.googleapis.com`、本地 → 指定的 baseURL）
> **送出前確認對話框**：對外送出時跳。可選「下次起不問」

LLM 連動**關著**也能用、只是沒徽章的列多一些。全部手動選也沒問題。

## 5. 匯入紀錄與整批沖銷

導覽列**「匯入紀錄」**進入 `ImportHistory` 畫面。歷次匯入按時序排。

每個 batch 列：

| 欄位 | 內容 |
|---|---|
| 匯入日時 | 什麼時候匯入的 |
| 解析器 | 用哪個 parser（楽天卡 等）|
| 檔名 | 原 CSV 檔名 |
| 件數 | batch 建立的傳票數 |
| 狀態 | `有效 N` / `全件沖銷` / `有效 X / 沖銷 Y`（部分沖銷）|

### 整批沖銷

點某列展開 → **「整批沖銷（{count} 件）」**按鈕。

> 確認對話框：「這個 batch 建立的有效傳票全部標記『已沖銷』。各傳票會建一張對應的沖銷傳票，兩邊都保留為歷史紀錄」

按下去一次產生所有沖銷傳票、batch 狀態變**「全件沖銷」**。原傳票保留為歷史（電子帳簿保存法要件）。

> 匯錯 CSV／已知側科目選錯時用這個一次倒回去、再重匯。

## 6. 自動分類規則

設定 → **「自動分類規則」**區塊登記**符合內容字串就自動填入對應科目**的規則。

### 6-1. 規則欄位

| 欄位 | 內容 |
|---|---|
| 比對方式 | `內容包含` / `交易對象名一致` / `正規表示式` |
| 比對字串 | 要比對的字串（例：`amazon`、`/^電気料金/`）|
| 對應科目 | 命中時要填的科目（例：`5200 消耗品費`）|
| 輔助科目（選填）| 同上的輔助科目 |
| 交易對象（選填）| 命中時順便填入的交易對象 ID |
| 優先度 | 數字大的規則先評估。多筆命中時優先度高的贏 |

### 6-2. 登記例

| 比對字串 | 對應科目 | 用途 |
|---|---|---|
| `amazon` | 5200 消耗品費 | Amazon 出費 → 消耗品費 |
| `AWS` | 5150 通信費（輔助：AWS）| AWS 每月帳單 |
| `ネットフリックス` | 5150 通信費 | 訂閱費 |
| `Adobe` | 5150 通信費 | Creative Cloud 月費 |
| 正規表示式 `/電気|TEPCO/` | 5130 水道光熱費 | 電力公司彙總 |

### 6-3. 規則在哪起作用

- **CSV 匯入時**：命中規則的列出**規則**徽章＋對應科目預填
- 命中規則的列**不問 LLM**（為了成本・速度・正確性）

> 規則愈多，CSV 匯入手動工作量趨近零。從常出現的交易開始登起。

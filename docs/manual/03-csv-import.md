# 03. CSV 取込

**Language**: **日本語** | [English](03-csv-import_en.md) | [繁體中文](03-csv-import_zh-TW.md)

銀行・クレジットカードの取引履歴 CSV から仕訳を一括作成する手順。

> **この章を読み終えたら出来ること**
> - 対応する銀行・カードの CSV をダウンロードして取り込める
> - 重複ファイルの検知が効くことを理解できる
> - 自動分類ルールで対方科目を自動入力できる
> - LLM 自動分類を使える
> - 取込履歴からバッチ単位で訂正（reverse）できる
>
> **前提**：[01. 初回設定](01-setup.md) を済ませてある。必要なら補助科目（カード別の `2120 未払金 / 楽天カード` 等）も登録済み。

## 1. 対応している銀行・カード

| 種別 | 対応 |
|---|---|
| 銀行 | 三菱UFJ／三井住友／SBI新生／PayPay（クレジット決済運用のみ、残高は未対応）|
| クレジットカード | 楽天／JCB（リクルートカード等含む）／セゾン／三井住友／三菱UFJ／au PAY／PayPay／ビュー（JRE CARD）／ライフ |

> すべて実 CSV で検証済みのパーサです。新しい銀行・カードを追加したい場合は [CONTRIBUTING.md](../../CONTRIBUTING.md) を参照（パーサーの追加方法）。

## 2. CSV をダウンロードする

各サービスの会員サイトから CSV を取得します。代表例：

| サービス | 取得方法の目安 |
|---|---|
| 三菱UFJ 銀行 | ダイレクトログイン → 入出金明細 → 期間指定 → CSV ダウンロード |
| 三井住友銀行 | SMBC ダイレクト → 入出金明細 → CSV |
| SBI新生銀行 | パワーダイレクト → 入出金照会 → CSV |
| 楽天カード | e-NAVI ログイン → 利用明細 → CSV ダウンロード |
| JCB / リクルートカード | MyJCB → ご利用明細 → CSV |
| セゾンカード | Net Answer → 利用明細 → CSV |
| 三井住友カード | Vpass → 明細 → CSV |
| 三菱UFJカード | 会員サイト → 明細 → CSV |
| au PAY カード | 会員サイト → 利用明細 → CSV |
| ビューカード | VIEW's NET → 利用明細 → CSV |
| ライフカード | LIFE-Web Desk → 利用明細 → CSV |

> **PayPay カード**：会員サイトの導線は時期によって変動。Web 版で CSV が出ないアカウントは OCR か手入力で。
>
> **エンコーディング**：パーサーが自動判定（Shift_JIS / UTF-8）。文字化けする場合は提出元の出力エンコーディングを確認。

## 3. 取り込みの 3 ステップ

ナビ 「**インポート**」 で `Import` 画面に移動。

### 3-1. ステップ 1：パーサーを選ぶ

「**1. パーサー**」 ドロップダウンから取引元を選択。選ぶと 「**既知側科目**」 が自動表示されます（例：`楽天カード` → `2120 未払金`）。

複数の口座やカードを補助科目で区別している場合は、「**補助科目（任意）**」 で対応する補助科目を選ぶ（例：`楽天カード`）。

> 既知側科目は CSV の各行に共通する仕訳の片側（カードなら未払金、銀行なら普通預金）。**対方科目**（経費・収益など）は次のステップで各行ごとに選びます。

### 3-2. ステップ 2：CSV ファイルを選ぶ

「**2. CSV ファイル**」 のフィールドからファイルを選択。

選択後、**自動でファイル内容を読み取り**、プレビューが下に表示されます。

#### 重複検知

aoiko はファイルの **ハッシュ**（中身の指紋）を記録していて、同じファイルを 2 回目以降にインポートしようとすると赤いバナー：
> このファイルは {date} に既にインポート済みです（{name}）

が表示され、誤って重複インポートするのを防ぎます。

> 同じ期間の CSV を再ダウンロードするとファイル内容が一致するため検知が効く。期間が一部重なる別 CSV（例：先月末〜今月末 vs 今月）はハッシュが違うため検知されません。重複行の手動確認が必要です。

### 3-3. ステップ 3：各行の対方科目を選ぶ

CSV の各行が **取込候補**としてテーブル表示されます。各行で：

| 列 | 内容 |
|---|---|
| 日付 / 摘要 / 金額 | CSV から読み取った値 |
| 対方科目 | この行の借方／貸方の **もう片方**（あなたが選ぶ）|
| 補助科目（任意）| 対方科目に補助科目があれば |
| 除外 | チェックでこの行はインポートしない |

#### 自動入力バッジ

ヘッダー上部に 「**{valid} / {total} 件 登録予定**」 が表示され、行ごとに次のいずれかのバッジが付きます：

| バッジ | 意味 |
|---|---|
| **規則** | 自動分類ルール（[6 節](#6-自動分類ルール) 参照）にマッチ |
| **LLM↑** | LLM が高信頼で対方科目を推定 |
| **LLM↓** | LLM の推定だが信頼度低い、人手確認推奨 |
| （バッジ無し）| 自動入力されず、手動で選ぶ必要あり |

各バッジは hover で詳細表示。LLM バッジは LLM 連携が有効な時のみ出ます。

#### 一括チェック

- バッジが **規則**なら基本的に信頼してそのまま
- **LLM↑** は内容を見て妥当か軽く確認
- **LLM↓** は必ず確認
- バッジ無しの行は **対方科目**を手動選択

### 3-4. 登録する

「**{count} 件登録**」 ボタンを押すと、確認なしで一気に仕訳化されます。

> 押す前に件数とバッジをよく確認。間違えても訂正仕訳で対応可能ですが、件数が多いと手間。
>
> 大量の誤登録に気づいたら、ナビ 「**履歴**」 から **バッチ単位で訂正**（[5 節](#5-取込履歴とバッチ訂正) 参照）が効率的。

成功すると上部に 「**{count} 件の仕訳を登録しました**」 が表示されます。

## 4. LLM 自動分類（任意）

[01. 初回設定 § 7](01-setup.md#7-ocrllm-を使う場合の準備) で **Gemini API キー** または **OpenAI 互換エンドポイント**を設定しておくと、CSV インポート時の自動分類で：

- **ルールにマッチしなかった行**を LLM が文脈で推定し、対方科目をプレフィル
- バッジ 「**LLM↑**」 / 「**LLM↓**」 が信頼度を表示

> **送信内容**：CSV 行のテキスト（金額・摘要等）＋ 勘定科目一覧
> **送信先**：選択中のエンジン（Gemini なら `generativelanguage.googleapis.com`、ローカルなら指定 baseURL）
> **送信前確認ダイアログ**：外部送信時のみ表示。「次回から確認しない」も選べます

LLM 連携を **OFF** のまま使うと、バッジ無し行が増えるだけ。手動で全部選んでも問題ありません。

## 5. 取込履歴とバッチ訂正

ナビ 「**履歴**」 で `ImportHistory` 画面に移動。過去のインポートが時系列で並びます。

各バッチ行：

| 列 | 内容 |
|---|---|
| インポート日時 | いつインポートしたか |
| パーサー | どのパーサーで（楽天カード等）|
| ファイル名 | 元の CSV ファイル名 |
| 件数 | バッチで作成した仕訳件数 |
| 状態 | `有効 N` / `全件訂正済み` / `有効 X / 訂正 Y`（部分訂正） |

### バッチ全件訂正

行を展開（クリック）→ 「**バッチを全件訂正（{count} 件）**」 ボタン。

> 確認ダイアログ：「このバッチで作成された有効な仕訳すべてを『訂正済み』にします。各仕訳に打消し仕訳が新しく作成され、両方とも履歴として保持されます」

押すと一括で打消し仕訳が生成され、バッチ状態は 「**全件訂正済み**」 に変わります。元の仕訳は履歴として残ります（電子帳簿保存法の要件）。

> CSV を間違って取り込んだ／間違った既知側科目で取り込んだ等の場合、これでまとめて巻き戻して、再取込できます。

## 6. 自動分類ルール

設定 → 「**自動分類ルール**」 セクションで、**取引内容文字列にマッチした行に対方科目を自動入力**するルールを登録できます。

### 6-1. ルールの形

| 項目 | 内容 |
|---|---|
| マッチ方式 | `内容に含む` / `取引先名一致` / `正規表現` |
| パターン | マッチさせる文字列（例：`amazon`、`/^電気料金/`）|
| 対方科目 | マッチした行に自動入力する科目（例：`5200 消耗品費`）|
| 補助科目（任意）| 同上の補助科目 |
| 取引先（任意）| マッチしたら同時に取引先 ID も入る |
| 優先度 | 数値が高いルールから順に評価。複数マッチしたら高優先度が勝つ |

### 6-2. 登録の例

| パターン | 対方科目 | 用途 |
|---|---|---|
| `amazon` | 5200 消耗品費 | Amazon の出費を消耗品費に |
| `AWS` | 5150 通信費（補助：AWS） | AWS の毎月の請求 |
| `ネットフリックス` | 5150 通信費 | サブスクを通信費に |
| `Adobe` | 5150 通信費 | Creative Cloud の月額 |
| `電気` (正規表現 `/電気|TEPCO/`) | 5130 水道光熱費 | 電力会社まとめて |

### 6-3. ルールが効く場面

- **CSV インポート時**：ルール命中行は **規則** バッジ＋対方科目プレフィル
- 自動分類ルールに命中した行は LLM に問い合わせない（コスト・速度・正確性のため）

> 規則が増えていけば、CSV インポート時の手作業がほぼゼロになります。よく出てくる取引から順に登録するのがおすすめ。

# 04. 領収書 OCR

紙のレシート・領収書画像から仕訳候補を作る。

**Language**: **日本語** | [English](04-receipt-ocr_en.md) | [繁體中文](04-receipt-ocr_zh-TW.md)

> **この章を読み終えたら出来ること**
> - 写真や PDF 画像から日付・店名・合計・適格請求書登録番号を抽出して仕訳化できる
> - 3 つの OCR エンジン（Gemini / OpenAI 互換 / Tesseract）の使い分けが分かる
> - クラウド送信前の確認ダイアログの意味と「次回から確認しない」の取り扱い方が分かる
>
> **前提**：[01. 初回設定 § 7](01-setup.md#7-ocrllm-を使う場合の準備) で OCR/LLM エンジンの設定を済ませてある。

## 1. エンジンの選び分け（要約）

| エンジン | 精度 | 端末外送信 | 必要なもの |
|---|---|---|---|
| **Gemini Vision**（既定）| ◎ | あり | Gemini API キー |
| **OpenAI 互換**（Ollama 等）| ◯〜◎ | localhost なら無し / リモートなら有り | エンドポイント＋ vision モデル |
| **Tesseract** | △ | 無し（traineddata 初回 DL のみ）| 設定で選択するだけ |

> 設定の詳細手順は [01. § 7](01-setup.md#7-ocrllm-を使う場合の準備) を参照。

## 2. 取込のステップ

ナビ **「領収書」** で `Receipt` 画面に移動。

### 2-1. 画像を選ぶ

**「1. 画像を選ぶ（カメラ撮影も可）」** のファイル入力をクリック。

- PC：ファイルダイアログから JPG / PNG / WebP / HEIC を選ぶ
- スマホ（PWA インストール済）：カメラ起動オプションが出るのでその場で撮影もできる

選ぶとプレビュー画像が下に表示されます。

> **撮影のコツ**：
> - 平らな場所、影が入らない角度
> - レシート全体が枠に収まるように
> - 蛍光灯の反射でテカらないように
> - 数千円の領収書なら 1〜2 MP 程度で十分。高解像度過ぎは LLM のレートリミット観点で逆効果

### 2-2. 解析する

**「解析する」** ボタンを押すと、選択中のエンジンに送って解析開始。

#### クラウドエンジン（Gemini / リモート OpenAI 互換）の場合：送信前確認ダイアログ

外部送信が伴うため、ボタンを押した直後に **CloudSendConfirmDialog** が出ます：

> 〇〇 へ画像を送信します。よろしいですか？
>
> ☐ 次回から確認しない

- 内容を確認して **「送信」** で実行
- **「キャンセル」** で中止
- 「次回から確認しない」にチェックを入れて送信すると、それ以降同じエンジンへの送信時の確認はスキップされます

> **チェック前に再確認**：「次回から確認しない」は IndexedDB の `skipExternalSendConfirm: true` に記録され、設定画面で個別に解除できません（全エンジン共通フラグ）。間違えてチェックしてしまった場合は「設定 → データ管理」から該当キーをリセット（または該当 IndexedDB を手動編集）。

#### Tesseract の場合：送信ダイアログ無し

WASM で端末内処理されるため確認ダイアログは出ません。代わりに解析中、`tesseract.js` が初回のみ traineddata（数 MB）を CDN から DL します。

### 2-3. 抽出結果の確認・修正

解析が終わると **「2. 抽出結果（編集可）」** セクションが下に展開されます：

| フィールド | 内容 |
|---|---|
| 取引日 | レシートの日付。`YYYY-MM-DD` |
| 店名 / 取引先 | 店舗名（編集可） |
| 合計金額（円） | 税込合計（編集可、整数）|
| インボイス登録番号 | T+13 桁が認識された時のみ表示（参考） |
| 内訳 | 品目があれば collapse で展開可（参考のみ、仕訳には反映されない）|

> **vision LLM 路 vs Tesseract 路の差**：
> - vision LLM は店名・合計をかなり高精度で抽出。品目内訳も拾ってくれる
> - Tesseract は **日付・合計・T+13 登録番号のみ** が確定性ベースで抽出される。**店名や品目は推定せず空のまま**。OCR 全文は内部処理用に残るが、仕訳の摘要には自動コピーされない

#### Tesseract のときの警告バナー

Tesseract 路で解析した場合、抽出結果ヘッダの下に黄色の注意バナーが出ます：

> ローカル OCR（Tesseract）の結果です。精度は限定的なため、合計金額・日付・店名は必ずご確認のうえ修正してください。

### 2-4. 対方科目・支払元を選ぶ

抽出結果の下部に 2 つのドロップダウン：

| ドロップダウン | 既定値 | 意味 |
|---|---|---|
| **対方科目（経費）** | `5910 雑費` | このレシートを計上する経費科目 |
| **支払元** | `1110 現金` | 何で払ったか（現金・普通預金・未払金 等）|

レシートが事業用・家計按分なしの単純経費なら、適切な経費科目に変更（例：飲食レシート → `5170 接待交際費`、文房具 → `5200 消耗品費`）。

カード払いなら **支払元** を `2120 未払金` に変更（補助科目を使っているなら該当カードへ）。

### 2-5. 登録する

**「仕訳を登録」** で確定。借方＝経費／貸方＝支払元の 2 行仕訳が作成され、ホームの直近一覧に出ます。

> 抽出結果に **インボイス登録番号** が入っていた場合、その仕訳行の `invoiceCompliant` フラグが `true` で保存されます（消費税本則課税で仕入税額控除の対象として扱える）。

## 3. 各エンジンの実用 tips

### Gemini Vision

- 無料枠で月数百枚は余裕。レシート画像はトークン消費少なめ
- 撮影時にレシートが少々斜めでも、影が部分的にあっても比較的強い
- 解像度は 1〜2 MP で十分。それ以上は速度・コスト的に逆効果

### OpenAI 互換（Ollama / LM Studio 等）

- **vision モデル必須**（テキスト専用モデルでは画像入力で失敗）
- 推奨モデル：`llama3.2-vision`、`ministral-3`、`moondream` 系
- Ollama 側：`OLLAMA_ORIGINS` に aoiko の URL（例 `http://localhost:31527`）を必ず追加
- aoiko 自体も `npm run preview` でローカル起動（HTTPS 配信版は localhost に接続不可）

### Tesseract

- **vision LLM が使えない／使いたくない**ときの選択肢
- 精度面で店名・品目はほぼ抽出できない前提で使う
- T+13 登録番号は確定性高い（正規表現で抽出するため）
- 日付・合計は当たればラッキー程度。必ず人手で確認・修正
- 完全オフライン運用したい場合は設定で `traineddata` を自己ホストへ

## 4. うまく抽出できない時のチェック

| 症状 | 対処 |
|---|---|
| 合計金額がずれる | 撮り直し or 手動修正。LLM 系なら通常少ない、Tesseract 系は頻発するので人手前提 |
| 日付が空 | 「YYYY/MM/DD」「令和○年」両形式試行。手動入力でも OK |
| 店名がローマ字混じり / 文字化け | LLM 系：撮影品質・解像度を上げる。Tesseract：そもそも店名抽出は推奨外なので諦めて手動 |
| インボイス番号が認識されない | 番号の印字部分を中心に明るく撮り直す。LLM 系は 1〜2 mm の小さい文字でも拾える |
| 接続エラー | 設定の **「接続テスト」** で API キー・エンドポイントを確認 |

## 5. プライバシー上の留意

- レシートに第三者の個人情報（顧客名・住所）が写る場合、cloud エンジンへの送信前に確認を
- 高機密の領収書（個人医療費・特殊事業の取引先 等）は Tesseract か Ollama on localhost を選ぶのが安心
- 詳細は [PRIVACY.md](../../PRIVACY.md) を参照

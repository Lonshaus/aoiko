# 05. 注文取込

從 Amazon・楽天 等 EC 網站訂單頁的貼上文字，建立有品項明細的傳票。

**Language**: [日本語](05-order-import.md) | [English](05-order-import_en.md) | **繁體中文**

> **讀完這章你能做到**
> - 貼上 EC 訂單頁文字，產生品項別的傳票候選
> - 在 1 張傳票內為每個品項指定不同的經費科目
> - 處理值引行（負金額）讓它走貸方
> - 確認・調整品項合計與訂單總額的差分
>
> **前提**：[01. 初次設定 § 7](01-setup_zh-TW.md#7-要用-ocrllm-才做的設定) 已設好 **Gemini API 金鑰** 或 **OpenAI 相容 endpoint**（Tesseract 不能用）。

## 1. 這個功能的意義

信用卡 CSV 只看到「楽天市場 ¥3,280」，要按品項分經費科目（書 → 新聞圖書費、消耗品 → 消耗品費 等）就要手動拆。

**注文取込**讓你貼上訂單頁全文、LLM 幫你抽出品項內訳。不是 DOM 爬取而是 LLM 解析、所以 Amazon / 楽天 等 UI 改版不影響。

## 2. 取込步驟

導覽列**「注文取込」**進入 `OrderImport` 畫面。

### 2-1. 取得訂單頁文字

開 EC 網站的訂單詳細頁（不是訂單一覧而是**個別訂單的詳細**）。例：

- **Amazon**：訂單歷史 → 單筆訂單的**「訂單詳細」**
- **楽天市場**：購入歷史 → 各訂單的詳細頁
- **Yahoo! ショッピング**：訂單歷史 → 點訂單號

頁面上 `Cmd+A`（Win: `Ctrl+A`）全選 → `Cmd+C` 複製。

> 頁眉・導覽・推薦商品・頁尾等雜訊一起進剪貼簿沒關係、LLM 會判別只抓需要的。不必神經質精準選範圍。

### 2-2. 貼上然後解析

aoiko 的**「1. 貼上訂單頁文字」**textarea 用 `Cmd+V` 貼。

按**「解析」**送到選中的 LLM 引擎解析。

#### 雲端引擎使用時

跟 [04. § 2-2](04-receipt-ocr_zh-TW.md#22-解析) 一樣會跳送出前確認對話框。「下次起不問」也可以在這裡設（跟收據 OCR 共用 flag）。

### 2-3. 抽取結果確認・修正

解析完**「2. 抽取結果（可編輯）」**展開。

#### 頭部欄位（可編輯）

| 欄位 | 內容 | 例 |
|---|---|---|
| 訂單日 | 訂單確定日 | `2026-05-20` |
| 交易對象 | 網站＋店舗 | `Amazon.co.jp` / `楽天市場 - ヨドバシ.com` |
| 訂單號 | 網站的訂單 ID（選填）| `250-1234567-1234567` |
| 合計金額（円）| 配送料・稅込・値引適用後的支付總額 | `4,580` |

#### 品項表格（可編輯）

每個品項 1 行：

| 欄 | 內容 |
|---|---|
| **品項** | 品項名（含型號・規格、可編輯）|
| **金額** | 單價×數量（整數、可編輯。值引行用負值 `-300` 等）|
| **科目** | 該品項要計上的經費科目（預設 `5200 消耗品費`、下拉選擇）|
| ✕ | 刪除該行 |

下方**「+ 新增行」**按鈕。LLM 抽漏的可手動補。

#### 支付來源

表格下方**「支付來源（預設：未払金）」**下拉。刷卡用 `2120 未払金`（有用輔助科目分卡的話選該卡），PayPay 餘額用 `1110 現金` 或另外設定的科目。

### 2-4. 品項合計與總額不一致警告

按下**「建立傳票」**瞬間，aoiko 算**品項合計與「合計金額」的差**。對不上就跳確認對話框：

> 品項合計與總額不一致（品項 ¥4,280 / 總額 ¥4,580）。要以總額為準繼續嗎？

- **OK**：總額側放支付來源 credit 1 行、品項側 debit。差額會反映在 credit (支付來源) − debit (品項) 之間
- **取消**：先修品項再重新建立

> 不一致的典型例子：LLM 漏抓「點數使用 -300 円」、配送料不在品項裡、消費稅的端數處理差異 等。「合計對不上但不想全改」的話按 OK 繼續、之後再用沖銷傳票補也行。

### 2-5. 建立的傳票結構

例：3 品項 + 配送料、合計 ¥4,580、刷卡：

```
2026-05-20  Amazon.co.jp 250-1234567-1234567
  借方  5200 消耗品費    ¥1,580   螺絲組
  借方  5200 消耗品費    ¥2,580   USB-C hub
  借方  5120 荷造運賃    ¥420     配送料
  貸方  2120 未払金     ¥4,580   合計
```

有值引行（負值）時、會自動跑到貸方：

```
2026-05-20  楽天市場 - ヨドバシ.com
  借方  5200 消耗品費    ¥3,000   商品 A
  貸方  5200 消耗品費    ¥500     優惠券折扣（同科目扣除）
  貸方  2120 未払金     ¥2,500   合計
```

> 傳票的 **source** 是 `paste`（[02. § 2-1](02-journal_zh-TW.md#2-1-篩選器) 想用 source 搜尋時用）。

## 3. 實用 tips

### 品項科目分類例

| 品項例 | 推薦科目 |
|---|---|
| 書・電子書 | `5xxx 新聞圖書費`（標準 seed 沒有的話手動加到 seed） |
| 文具 / 線材 / USB hub | `5200 消耗品費` |
| 業務用 PC・螢幕 | `1514 工具器具備品` → 另外當固定資產登錄（[08. 減價償卻](08-depreciation_zh-TW.md)）|
| AWS / SaaS 月費 | `5150 通信費`（輔助：服務名）|
| 配送料 | `5120 荷造運賃` or 併入商品科目 |
| 優惠券折扣 | 跟對應商品同科目用負值（或 `4910 雑収入` 用正值）|

### 貼上的小撇步

- **個別訂單詳細**頁貼上（訂單一覧頁有多筆訂單會讓 LLM 混亂）
- 1 次取込 = 1 訂單。LLM 已指示只抽第一筆
- 頁長（雜訊多）OK，但要確認訂單摘要部分有包含

### Amazon Business / 楽天 Business

法人會員網站有時可直接下載**「訂單歷史 report CSV」**。能拿到的話可能比注文取込＋卡片明細對照更可靠。aoiko 目前沒有 Amazon Business / 楽天 Business 專用 parser，要的話手動每行登錄。

## 4. 抽不到時的對策

| 症狀 | 對策 |
|---|---|
| 按「解析」沒反應 | 設定 →「LLM 連動」→「連線測試」確認 API 金鑰・endpoint |
| 結果不是中文/日文 | prompt 是日文、本來不應發生。本地 LLM 看模型選擇、換到支援日文的模型 |
| 品項抽漏 | LLM 限制。用**「+ 新增行」**手動補 |
| 金額錯 | 編輯該行。跟總額對不上就在確認對話框選 OK 或再修 |
| 訂單號空 | 選填、無視即可。傳票摘要只會放店名 |
| 配送料・手續費沒進品項 | 用「+ 新增行」加配送料行（指定金額・科目）|

## 5. 隱私注意

- 訂單頁通常有**配送地址・姓名・電話**。送雲端 LLM 前最後確認
- 自宅住址混進去的話用 localhost 的 Ollama、或把 textarea 個資行手動刪掉再按「解析」比較安全
- 詳見 [PRIVACY_zh-TW.md](../../PRIVACY_zh-TW.md)

## 6. 為什麼不是 Chrome 擴充

aoiko 當初 Phase 3 有考慮過「用 Chrome 擴充爬 Amazon / 楽天 歷史」，但下面 3 個原因改成「貼上 → LLM 抽取」：

1. **抗 DOM 改版**：EC 站 UI 改版時 scraper 會壞、長期維護負債大
2. **避開擴充權限・發布的摩擦**：使用者心理門檻、Chrome Web Store 配信成本
3. **行動端對應**：擴充功能在手機 Chrome 不能跑

LLM 抽取方式對 DOM 改版有耐性、行動端含全環境都能跑、現有 LLM Adapter（Gemini / Ollama）能直接再利用。

# 05. 注文取込

Amazon・楽天 等 EC サイトの注文ページの貼り付けテキストから、品目内訳付きの仕訳を作る。

**Language**: **日本語** | [English](05-order-import_en.md) | [繁體中文](05-order-import_zh-TW.md)

> **この章を読み終えたら出来ること**
> - EC 注文ページのテキストを貼り付けて、品目別の仕訳候補を生成できる
> - 各品目に個別の経費科目を割り当てて 1 仕訳にまとめられる
> - 値引行（負値）を貸方に振り分けて処理できる
> - 品目合計と注文総額の差分を確認・調整できる
>
> **前提**：[01. 初回設定 § 7](01-setup.md#7-ocrllm-を使う場合の準備) で **Gemini API キー** または **OpenAI 互換エンドポイント**を設定済み（Tesseract は使えません）。

## 1. この機能の意義

カード CSV は「楽天市場 ¥3,280」までしか分からないため、品目で経費科目を分けたい場合（書籍 → 新聞図書費、消耗品 → 消耗品費 等）に手作業が必要でした。

**注文取込**は注文ページ全文を貼り付けると LLM が品目内訳を抽出してくれる機能です。

## 2. 取込のステップ

ナビ **「注文取込」** で `OrderImport` 画面に移動。

### 2-1. 注文ページのテキストを取得

EC サイトの注文詳細ページ（注文一覧ではなく **個別注文の詳細**）を開きます。例：

- **Amazon**：注文履歴 → 個別注文の **「注文の詳細」**
- **楽天市場**：購入履歴 → 各注文の詳細ページ
- **Yahoo! ショッピング**：注文履歴 → 注文番号クリック

ページ上で `Cmd+A`（Win: `Ctrl+A`）で全選択 → `Cmd+C` でコピー。

> ヘッダ・ナビ・レコメンド商品・フッタ等の不要部分が混ざっていても、LLM が判別して必要な箇所だけ拾います。神経質に範囲選択しなくて OK。

### 2-2. 貼り付けて解析

aoiko の **「1. 注文ページのテキストを貼り付け」** の textarea に `Cmd+V`。

**「解析する」** ボタンを押すと、選択中の LLM エンジンに送信して解析。

#### クラウドエンジン使用時

[04. § 2-2](04-receipt-ocr.md#22-解析する) と同じく、送信前確認ダイアログが出ます。「次回から確認しない」もここで設定可能（領収書 OCR と共通フラグ）。

### 2-3. 抽出結果の確認・修正

解析が終わると **「2. 抽出結果（編集可）」** が展開されます。

#### ヘッダ部（編集可能）

| フィールド | 内容 | 例 |
|---|---|---|
| 注文日 | 注文確定日 | `2026-05-20` |
| 取引先 | サイト＋店舗 | `Amazon.co.jp` / `楽天市場 - ヨドバシ.com` |
| 注文番号 | サイトの注文 ID（任意） | `250-1234567-1234567` |
| 合計金額（円） | 配送料・税込・値引適用後の支払総額 | `4,580` |

#### 品目テーブル（編集可能）

各品目が 1 行：

| 列 | 内容 |
|---|---|
| **品目** | 品目名（型番・規格含む、編集可）|
| **金額** | 単価×数量（整数、編集可。値引行は負値 `-300` 等）|
| **科目** | この品目を計上する経費科目（既定 `5200 消耗品費`、ドロップダウンで変更）|
| ✕ | 行を削除 |

下に **「+ 行追加」** ボタン。LLM が拾い漏れた品目を手動で追加できます。

#### 支払元

テーブル下に **「支払元（既定：未払金）」** ドロップダウン。クレジットカード払いなら `2120 未払金`（補助科目で各カード）、PayPay 残高なら `1110 現金`（または別途設定の科目）等。

### 2-4. 品目合計と総額のミスマッチ警告

**「仕訳を登録」** を押した瞬間、aoiko は **品目の合計と「合計金額」の差**を計算します。一致しないと確認ダイアログ：

> 品目合計と総額が一致しません（品目 ¥4,280 / 総額 ¥4,580）。総額側を信用して続行しますか？

- **OK**：総額側を支払元への 1 行 credit に使い、品目側は debit。差額は「支払元の credit 額 − 品目 debit 合計」として現れる
- **キャンセル**：品目側を修正してから再度登録

> ミスマッチの典型例：LLM が「ポイント利用 -300 円」を拾い損ねた、配送料が品目に入っていない、消費税の端数処理が違う 等。「合計が違うけど品目を全部直す気力は無い」場合は OK で続行して、後で訂正仕訳でもよい。

### 2-5. 登録される仕訳の構造

例：3 品目 + 配送料、合計 ¥4,580、クレジット払い

```
2026-05-20  Amazon.co.jp 250-1234567-1234567
  借方  5200 消耗品費    ¥1,580   ねじ穴セット
  借方  5200 消耗品費    ¥2,580   USB-C ハブ
  借方  5120 荷造運賃    ¥420     配送料
  貸方  2120 未払金     ¥4,580   合計
```

値引行（負値）がある場合は credit 側へ：

```
2026-05-20  楽天市場 - ヨドバシ.com
  借方  5200 消耗品費    ¥3,000   商品 A
  貸方  5200 消耗品費    ¥500     クーポン値引（同科目から控除）
  貸方  2120 未払金     ¥2,500   合計
```

> 仕訳の **source** は `paste`（[02. § 2-1](02-journal.md#2-1-絞り込みフィルタ) で source 検索したい時用）。

## 3. 実用 tips

### 品目の科目分けの実例

| 品目例 | 推奨科目 |
|---|---|
| 書籍・電子書籍 | `5xxx 新聞図書費`（標準 seed に無ければ手動で seed に追加が必要）|
| 文房具 / ケーブル / USB ハブ | `5200 消耗品費` |
| 業務 PC・モニタ | `1514 工具器具備品` → 固定資産として別途登録（[08. 減価償却](08-depreciation.md)）|
| AWS / SaaS 月額 | `5150 通信費`（補助：サービス名）|
| 配送料 | `5120 荷造運賃` or 商品科目に含める |
| クーポン値引 | 該当商品と同じ科目に負値で（または `4910 雑収入` に正値） |

### 注文ページの貼り付けコツ

- **個別注文の詳細**を貼り付ける（注文一覧ページだと複数注文が混在して LLM が混乱する）
- 1 回の取り込みは **1 注文**まで。LLM は最初の注文のみ抽出するよう指示されています
- 長いページ（HTML レコメンド多数）も大丈夫だが、注文サマリ部分を含めていることを確認

### Amazon Business / 楽天 Business

法人向けの会員サイトでは「**注文履歴レポート CSV**」を直接 DL できる場合があります。それが取得できるなら、注文取込より [03. CSV 取込](03-csv-import.md) でカード明細とは別レイヤーで取り込む方が漏れが少ないこともあります（aoiko には現状 Amazon Business / 楽天 Business 専用 parser は無く、必要なら手動で行ごとに登録）。

## 4. うまく抽出できない時のチェック

| 症状 | 対処 |
|---|---|
| 「解析する」を押しても進まない | 設定 → 「LLM 連携」→ 「接続テスト」で API キー・エンドポイントを確認 |
| 解析結果が日本語じゃない | プロンプトは日本語なので通常起きないが、ローカル LLM のモデル選択次第。日本語サポートのモデルへ切替 |
| 品目が抜けている | LLM 抽出の限界。**「+ 行追加」**で手動補完 |
| 金額が違う | 該当行を編集。総額と合わなければ確認ダイアログで OK か再修正か選ぶ |
| 注文番号が空 | 任意フィールドなので無視可。仕訳の摘要には店名のみ入る |
| 配送料・手数料が品目に入っていない | 「+ 行追加」で配送料行を追加（金額・科目を指定） |

## 5. プライバシー上の留意

- 注文ページには **配送先住所・氏名・電話番号** が含まれることが多い。cloud LLM への送信前に最終確認を
- 自宅住所が混入する場合は localhost の Ollama を使うか、textarea から手動で個人情報の行を削除してから「解析する」を押すのが安全
- 詳細は [PRIVACY.md](../../PRIVACY.md) を参照

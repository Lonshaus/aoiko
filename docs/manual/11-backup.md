# 11. バックアップと復元

File System Access API、OPFS、手動 JSON ダウンロード／復元の運用。

**Language**: **日本語** | [English](11-backup_en.md) | [繁體中文](11-backup_zh-TW.md)

> **この章を読み終えたら出来ること**
> - バックアップフォルダを指定して自動 JSON バックアップを有効化できる
> - 手動で JSON ファイルをエクスポートできる
> - バックアップから復元できる
> - データ消失リスクと対策を理解できる
>
> **前提**：[01. 初回設定](01-setup.md) 済み。記帳をしてデータが蓄積されている状態。

## 1. なぜバックアップが必要か

aoiko のデータはブラウザの **IndexedDB**（端末ローカル）に保存されます。これは：

- ✅ サーバを介さず外部に漏れない（プライバシー）
- ❌ ブラウザのサイトデータ削除 → **完全消失**
- ❌ デバイス故障 → 復元不能
- ❌ ブラウザのプロファイル削除 → 失う

定期的なバックアップは **利用者責任**です。「気付いた時に取る」運用は事故の元なので、自動バックアップを推奨。

## 2. バックアップ方式の比較

| 方式 | API | 対応ブラウザ | 推奨度 |
|---|---|---|---|
| **File System Access API（FSA）** | `showDirectoryPicker` | Chrome / Edge / Brave 等 Chromium 系 | ◎ 自動・任意フォルダ |
| **OPFS（Origin Private File System）** | `navigator.storage.getDirectory` | Safari / Firefox 含む大半 | ◯ 自動だがブラウザ管理領域 |
| **手動 JSON ダウンロード** | `<a download>` | 全ブラウザ | △ 思い出した時だけ |

aoiko は FSA が使えるブラウザでは FSA、使えなければ OPFS、それも使えなければ手動 DL のみ、と自動 fallback します。

## 3. 自動バックアップを設定（推奨）

設定 → **「バックアップ」** セクション。

### 3-1. Chromium 系（FSA 対応）

1. **「バックアップフォルダを選択」** をクリック
2. ブラウザがフォルダ選択ダイアログを開く
3. **任意の場所**（例：`Documents/aoiko-backup/`、Google Drive の同期フォルダ、Dropbox 内 等）を選択
4. アクセス権限のダイアログで **「許可」**
5. 設定画面に **「現在のフォルダ：〇〇」** と表示されれば成功

これ以降、仕訳の追加・修正のたびに `aoiko-ledger-{年月日時}.json` 形式で自動的にバックアップが書き出されます。

> **Google Drive / iCloud / Dropbox 連携の Tip**：FSA で指定したローカルフォルダがクラウド同期対象なら、実質クラウドバックアップになる。例：`~/Google Drive/My Drive/aoiko-backup/` を指定 → ローカルに書き出された JSON が自動で Google Drive に同期。
>
> **注意**：iCloud Drive の Finder 「ダウンロードオンデマンド」アイテムを FSA フォルダにすると、バックアップ書込時に毎回オンライン同期が走るので、容量に余裕のあるローカル領域を選ぶ方が安定。

### 3-2. Safari / Firefox（OPFS のみ）

FSA に未対応のブラウザでは、選択肢が **OPFS** のみ。OPFS は **ブラウザが内部管理する非公開ストレージ**で、ファインダーや Explorer から直接見ることはできません。

OPFS のバックアップは：
- 自動で書込される（FSA と同じく仕訳更新の都度）
- ブラウザのサイトデータ削除で **OPFS も一緒に消える**（IndexedDB と同じ運命）
- バックアップとしての本来の役割（端末/ブラウザ独立）は **果たさない**

> Safari / Firefox 利用者には **手動 JSON ダウンロード**との併用を強く推奨。

### 3-3. 最終バックアップ時刻の確認

設定 → 「バックアップ」セクションに **「最終バックアップ：2026-05-26 14:23」** 等が表示されます。長期間更新されていない場合は手動エクスポートで補完を。

## 4. 手動 JSON エクスポート

設定 → **「データ管理」** または同等メニューから **「JSON エクスポート」** を実行：

- 全データ（仕訳、補助科目、取引先、固定資産、設定 等）が 1 つの JSON ファイルにまとめられる
- ブラウザの「ダウンロード」フォルダに保存
- ファイル名：`aoiko-ledger-{タイムスタンプ}.json` 等

> **API キー・申告者情報は既定で除外されます**。「バックアップに API キーを含める」「申告者情報を含める」のトグルをそれぞれオンにしない限り、クラウド同期フォルダ等へ平文の API キーや個人情報が書き出されることはありません。別端末への移行等でこれらも一緒に持ち出したい場合のみオンにしてください。

このファイルを：

- 別の物理ストレージ（外付け SSD、USB メモリ）にコピー
- メールで自分宛て送信
- クラウドストレージに保存

…等して **複数箇所に分散**保管すると堅牢。

> 月末・四半期末・年度末等の **節目で手動 DL** する運用が、自動バックアップに加えて安心。

## 5. JSON から復元

### 5-1. 復元のシナリオ

- ブラウザのキャッシュをうっかり全削除した
- 新しい PC に移行したい
- ブラウザを変えたい（Chrome → Safari 等）

### 5-2. 復元手順

1. 設定 → **「JSON から復元」** セクション
2. **「ファイルを選択」** で JSON を指定
3. 内容サマリが表示される：
   > version 1 ・ 12 テーブル ・ 5,432 行
4. **「全データを置換して復元」** をクリック
5. 確認ダイアログ：
   > JSON で全データを置換しますか？
   > 現在の全データが削除され、選択した JSON の内容で完全置換されます。この操作は元に戻せません。
6. **「置換して復元」** で実行
7. 復元成功メッセージ → **「再読み込み」** でアプリ全体をリロード

### 5-3. 復元時の注意

- **完全置換**：現在の IndexedDB は丸ごと上書きされる。誤って実行したら元に戻せない
- **進行中の作業がある場合は先に手動エクスポート**してから復元する習慣を
- 復元後はレポート画面で **BS の整合性チェック**（[06. § 4-1](06-reports.md#4-1-不整合警告)）必須

## 6. 全データ削除（注意）

設定 → 「データ管理」→ **「全データ削除」**：

- IndexedDB のすべてのデータを物理削除（バックアップファイルは保持）
- 警告ダイアログ後に実行
- 削除後は初期状態に戻り、再度免責同意・基本情報入力からスタート

> **どんな時に使う？**：別のテストデータでクリーンスタートしたい、aoiko を譲渡したい、等の限定的ケース。常用しない。

## 7. バックアップ戦略の推奨

3 層構成：

| 層 | 目的 | 実装 |
|---|---|---|
| **層 1**：常時自動 | 短期保護（不慮の操作ミス対応）| FSA でクラウド同期対象フォルダを指定 |
| **層 2**：節目手動 | 中期保護（月末・年度末等のスナップショット）| 手動 JSON エクスポート → 別場所に保存 |
| **層 3**：定期確認 | 健全性確認 | 半年に一度、復元テストを別ブラウザのプロファイルで実施 |

> 申告完了後の年度は特に重要。年度ロック（[06. § 8](06-reports.md#8-年度ロック申告済み)）と併用すると変更検知も効く。

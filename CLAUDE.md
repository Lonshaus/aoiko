# Project Guidelines: PJ_aoiko

Pure-frontend bookkeeping tool for Japanese sole proprietors (青色申告 75万円 deduction from 令和 9 年分: requires e-Tax + 優良な電子帳簿; 65万円 deduction still supported). BYOK, no backend.

## Stack

Svelte 5 + TypeScript + Vite (no SvelteKit, pure SPA) · bits-ui + shadcn-svelte + Tailwind · Dexie · Decimal.js · pnpm · Node 22 LTS · VS Code

## Hard Rules

- **Money**: `Decimal.js` only. JS `number` for monetary values is forbidden. Convert to string at the Dexie boundary.
- **Svelte 5 runes only**. Forbidden Svelte 4 legacy: `$:`, `export let`, `on:` directives, `<slot>`/`let:`, `createEventDispatcher`, `beforeUpdate`/`afterUpdate`, `writable`/`readable`/`derived`/`get` from `svelte/store`, `$store` auto-subscription.
- **Allowed `$` auto-subscription exception**: Dexie's `liveQuery` (returns a Svelte store).
- **Global state**: class + singleton in `.svelte.ts` files using `$state` / `$derived`. Domain data (entries, vendors, assets) lives in Dexie, not app state.
- **`$effect` is an escape hatch**. Prefer `$derived`. Never set `$state` inside `$effect`.
- **Props**: `let { x }: Props = $props()`. Callback props (`onsubmit`) replace dispatched events. Snippets replace slots.
- **TypeScript strict** + `noUncheckedIndexedAccess` + `exactOptionalPropertyTypes`.
- **Use single quotes** for TS/JS string literals only (`.ts` / `.js` / `<script>` blocks inside `.svelte`). Other files follow their language convention: HTML / Svelte template attributes use double quotes, JSON requires double quotes by spec. Exception: shadcn-svelte vendor files (`src/lib/components/ui/**`, `src/lib/utils.ts`) keep their original style since they get regenerated by the CLI; don't reformat them by hand.
- **No comments** unless the WHY is non-obvious. Don't restate WHAT.
- **Code comments are written in Japanese** (target users and domain terminology are Japanese). User-facing strings are also Japanese. Identifiers stay English.
- `if`, `else`, `for`, `while` must always use `{}`, even for single-line bodies. The body must be on a new line, never on the same line as the opening `{`. `else` must be cuddled with the preceding closing brace on the same line: `} else {` or `} else if (...) {`, never on its own line.
- **No trailing blank lines**. Files must end with the last content character — no terminating LF (the visual empty line VS Code shows after the final `\n` counts as a trailing blank line). No redundant blank lines anywhere.
- **No blank line immediately before a comment.** Comments are attached to the code they describe. Even section-divider comments are written without a leading blank line. Applies to `//`, `/* */`, `/** */`, `<!-- -->`, `#`, etc. Consecutive comment lines do not need blank lines between them either.

## Domain Rules

- **Journal entries are the single source of truth.** CSV imports, extension scrapes, OCR results are *candidates* until user confirms.
- **Confirmed entries are immutable.** Corrections are reversing entries (修正仕訳). Preserve full audit history (電子帳簿保存法).
- **Year is a first-class dimension.** Tax law, account codes, and `.xtx` schema change yearly. Isolate year-bound logic (e.g. `src/tax-schema/2026/`).
- **Domain logic is framework-agnostic** pure TypeScript — testable with Vitest, no browser needed.

## Testing Requirements (mandatory)

- **Every new pure-logic module ships with co-located `*.test.ts`**: `lib/`, `domain/`, `parsers/`, `backup/payload.ts`, `tax-schema/*` — no exceptions.
- **Every new component with non-trivial logic ships with `*.test.svelte.ts` or component test.** Pure presentational shells without state are exempt.
- **DB-dependent tests use `fake-indexeddb`** (loaded via `vitest.setup.ts`). Do not mock Dexie; exercise the real Dexie API against the in-memory IDB.
- **`pnpm run test` must pass before considering any feature done.** When refactoring, update tests in the same change.
- **CSV parsers must include fixture-based snapshot tests**: anonymized real CSV input + expected `JournalEntry[]` JSON.
- Keep tests focused on observable behavior, not implementation detail. Don't test private getters; do test edge cases (empty, one-sided, negative, unicode, encoding boundaries).

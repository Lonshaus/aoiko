# セキュリティポリシー

**Language**: **日本語** | [English](SECURITY_en.md) | [繁體中文](SECURITY_zh-TW.md)

aoiko は純フロントエンドの BYOK（Bring Your Own Key）アプリです。サーバを持たず、利用者の機微データはすべてブラウザ内に閉じます。本書では既知のリスク・サポート方針・脆弱性報告手順を整理します。

## サポート対象バージョン

`master` ブランチの最新コミットのみサポートします。リリースタグは未運用です（Phase 4 完了後に開始予定）。

## 脆弱性の報告

機密性のある脆弱性報告は **GitHub Security Advisories** をご利用ください。

1. リポジトリの **Security** タブ → **Report a vulnerability**
2. 影響範囲・再現手順・想定される影響を記載
3. パブリック issue では報告しないでください

公開状態の問題（例：誤った勘定科目、UI バグ）は通常の issue で構いません。

返信時間は努力目標として 7 日以内を目指しますが、ボランティアベースのため保証はありません。

## 設計上のセキュリティ前提

### BYOK モデル

- 利用者が選択した OCR/LLM エンジン（Google Gemini API ／ OpenAI 互換 ／ Tesseract）の API キー・エンドポイント設定は **利用者自身が登録・自分のブラウザ IndexedDB に保管** します
- 開発者・配布者は利用者の API キー・エンドポイント情報を **取得・転送・保管しません**
- 外部 API 利用時のリクエストは **利用者のブラウザから直接、選択中のエンドポイントへ送信** されます（プロキシ無し）。Tesseract 選択時は LLM API 送信そのものが発生しません

### ストレージ

- 帳簿データ、API キー、設定はすべて **IndexedDB（端末のローカル）** に保存
- バックアップ：File System Access API（Chromium）／ OPFS（Safari, Firefox）／ 手動 JSON ダウンロード
- **aoiko の管理サーバへの送信は一切ありません**（aoiko には管理サーバが存在しません）。LLM/OCR API 利用時は利用者が設定した外部エンドポイント（Gemini / OpenAI 互換等）にのみ送信されます

## 既知のリスク

### 1. サーバ側監査ログの不在

- 不正アクセス・データ漏洩の **検知手段がありません**
- 端末侵害＝データ漏洩 と等価

### 2. ブラウザストレージ漏洩

- 同じ端末の別ユーザ、マルウェア、ブラウザ拡張機能から IndexedDB が読まれる可能性
- 個人情報・取引履歴・API キーがそのまま読まれます
- 業務端末でのご利用、端末暗号化を推奨します

### 3. LLM API 送信内容のリスク

- CSV 行・領収書画像は、利用者が選択中のエンジンに従って以下へ送信されます：
  - **Gemini** → `generativelanguage.googleapis.com`（Google のデータ取り扱い方針に従い、学習利用の有無は契約形態による）
  - **OpenAI 互換**（Ollama 等） → 利用者が指定した baseURL。localhost 指定時は端末外送信なし
  - **Tesseract** → 送信なし（WASM で端末内処理。traineddata 初回 DL のみ）
- 機密度の高いデータの送信前に確認してください（外部エンジン使用時は送信直前に確認ダイアログを表示）
- LLM/OCR 機能は **任意起動**（UI ボタン）であり、自動送信はしません

### 4. PWA キャッシュ

- 古いビルドが Service Worker にキャッシュされる可能性があります
- バグ修正版の伝播に時間差が出ます

## ハードニング推奨

- 端末のディスク暗号化を有効化
- ブラウザのプロファイル分離（業務用と私用を分ける）
- 信用しないブラウザ拡張機能をインストールしない
- 定期的なバックアップ
- 不要になった API キーは Google 側で **必ず失効** させる

- `npm audit` を CI で実行する予定（未実装）
- 緊急性の高い CVE は反映を急ぎますが、対応は保証されません
